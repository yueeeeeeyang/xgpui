# AGENTS.md

## 项目定位

`xgpui` 是一个基于 `gpui` 的 Rust UI 库，目标是沉淀稳定、可复用、结构清晰的基础 UI 组件。

本文件适用于当前目录及其所有子目录。后续如果在子目录中新增更具体的 `AGENTS.md`，则子目录内的规则以更近层级的文件为准，但不能削弱本文件中的强制约束。

## 强制约束

1. 当需求不完整、存在关键歧义，或继续实现会影响公开 API、组件行为、数据结构、交互方式、依赖选择、兼容性策略时，必须先向用户提问并确认，不能擅自补全关键产品或技术决策。

2. 所有新增或修改的文件、方法和逻辑代码，必须添加完整、详细、有效的中文注释。注释需要说明代码意图、关键设计原因、状态流转、边界条件和不易理解的实现细节，不能只写空泛描述。

3. 优先使用稳定版 Rust，不引入 nightly 特性、不依赖 nightly-only API、不要求用户切换 nightly 工具链。除非用户明确同意，否则不得使用 `#![feature(...)]` 或其他 nightly 专属能力。

4. 开发新功能或扩展现有功能时，必须遵守代码结构最佳实践。应按模块、类型、职责和复用边界拆分实现，避免把所有代码堆在同一个文件、同一个类型或同一个方法中。

5. 所有新增组件或扩展现有组件时，必须同步支持明暗两种皮肤。组件样式应接入 `foundation::theme` 提供的主题 token 和 `ThemeMode::Light`、`ThemeMode::Dark`，不能只为单一皮肤硬编码颜色；示例和文档也应同步体现明暗皮肤下的使用方式。

## 开发规范

1. 修改代码前，必须先阅读相关模块、类型、测试和已有约定，理解当前结构后再动手。

2. 优先遵循 Rust 生态常规组织方式，包括清晰的模块边界、明确的可见性控制、合理的错误处理、可读的类型定义和可维护的测试结构。

3. 公共 API、组件参数、组件状态、事件回调、布局逻辑、渲染逻辑、边界行为和兼容性约束，都需要通过中文注释说明设计意图。

4. 组件实现应保持职责单一。通用逻辑应提取为合适的辅助函数、模块、trait 或类型，但不能为了抽象而抽象。

5. 新增依赖前必须确认必要性，并优先选择稳定、维护良好、与 Rust 和 `gpui` 生态兼容的依赖。不能为了局部便利引入过重依赖。

6. 不得随意重构与当前任务无关的代码。若发现需要较大范围调整，应先说明原因、影响范围和替代方案。

7. 修改完成后，应优先运行适用的 Rust 检查或测试，例如 `cargo fmt --check`、`cargo check`、`cargo test`。如果当前项目尚未初始化或缺少可运行命令，需要在结果中明确说明。

## 注释要求

1. 中文注释必须服务于理解和维护，不能简单重复代码表面含义。

2. 对外暴露的类型、函数、方法、trait、组件和模块，应优先使用 Rust 文档注释说明用途、参数、返回值、行为约束和使用注意事项。

3. 对复杂逻辑、状态机、布局计算、事件分发、样式合成、生命周期处理和性能敏感路径，应在关键位置添加行内或块级中文注释解释原因。

4. 当代码存在非显而易见的取舍时，必须在注释中说明为什么采用当前实现，以及未来修改时需要注意的约束。

## 推荐代码结构

项目初期优先使用单 crate、分层模块结构，保持模块边界清晰、公共 API 稳定、组件职责单一。

推荐的基础目录结构如下：

```text
xgpui/
├── AGENTS.md
├── Cargo.toml
├── README.md
├── examples/
│   ├── button.rs
│   ├── input.rs
│   └── kitchen_sink.rs
├── src/
│   ├── lib.rs
│   ├── prelude.rs
│   ├── foundation/
│   │   ├── mod.rs
│   │   ├── color.rs
│   │   ├── spacing.rs
│   │   ├── radius.rs
│   │   ├── size.rs
│   │   └── theme.rs
│   ├── primitives/
│   │   ├── mod.rs
│   │   ├── text.rs
│   │   ├── icon.rs
│   │   ├── surface.rs
│   │   └── stack.rs
│   ├── components/
│   │   ├── mod.rs
│   │   ├── button/
│   │   │   ├── mod.rs
│   │   │   ├── props.rs
│   │   │   ├── style.rs
│   │   │   └── tests.rs
│   │   ├── input/
│   │   │   ├── mod.rs
│   │   │   ├── props.rs
│   │   │   ├── state.rs
│   │   │   ├── style.rs
│   │   │   └── tests.rs
│   │   └── checkbox/
│   │       ├── mod.rs
│   │       ├── props.rs
│   │       ├── style.rs
│   │       └── tests.rs
│   ├── interaction/
│   │   ├── mod.rs
│   │   ├── focus.rs
│   │   ├── keyboard.rs
│   │   └── mouse.rs
│   └── utils/
│       ├── mod.rs
│       └── id.rs
└── tests/
    └── public_api.rs
```

各目录职责如下：

1. `foundation/` 存放设计基础能力，例如颜色、间距、圆角、尺寸和主题 token。该目录不应放入具体业务组件。

2. `primitives/` 存放低层 UI 原语，例如文本、图标、容器和基础布局能力。原语应保持小而稳定，便于被多个组件组合复用。

3. `components/` 存放对外使用的完整组件，例如 `Button`、`Input`、`Checkbox`。复杂组件必须使用独立目录组织，不能把参数、状态、样式和渲染逻辑全部堆在单个文件中。

4. `interaction/` 存放焦点、键盘、鼠标等通用交互逻辑，避免相同交互规则散落在多个组件实现中。

5. `utils/` 只存放非常通用且不属于 UI 语义的辅助能力。新增工具函数前需要确认它确实有跨模块复用价值。

6. `prelude.rs` 集中导出最常用的公共类型，便于用户通过 `use xgpui::prelude::*;` 使用库能力。

7. `lib.rs` 只负责模块声明和公共 re-export，不应堆放具体组件实现、样式计算或业务逻辑。

复杂组件目录优先采用以下结构：

```text
components/button/
├── mod.rs
├── props.rs
├── style.rs
└── tests.rs
```

组件目录内文件职责如下：

1. `mod.rs` 作为组件主入口，负责组织子模块、定义或导出组件主体以及必要的公共类型。

2. `props.rs` 存放组件参数、变体、尺寸、状态输入等数据模型。

3. `state.rs` 仅在组件确实存在独立状态管理时添加，用于封装状态结构和状态流转规则。

4. `style.rs` 存放样式解析、主题映射、状态样式合成和视觉变体规则。

5. `tests.rs` 存放该组件自身的行为测试、样式决策测试或公共 API 约束测试。

第一阶段优先建设 `foundation/`、`primitives/`、`components/button/` 和 `prelude.rs`，以保证结构不过度设计，同时为后续组件扩展留下清晰边界。

## 质量要求

1. 新功能应包含与风险匹配的测试。涉及公共 API、组件行为、状态变化或边界条件时，必须补充对应测试或说明无法测试的原因。

2. 代码应尽量保持小步、聚焦、可审查。每次变更应围绕明确目标展开，避免混入无关格式化、无关重命名或无关重构。

3. 对用户可见的组件行为，应优先保证稳定性、可预测性和可组合性。

4. 任何可能破坏兼容性的改动，都必须先明确影响范围，并在用户确认后再实施。
